# CAD Export File Specification

## 1. Purpose

This document defines the **first export target for CAD drawings generated by RackEditor**.

The goal is to produce a file that:

- can be opened by common CAD systems,
- preserves the current 2D warehouse layout accurately,
- follows established CAD conventions where standards exist,
- and, where no standard fully defines the visual result, follows the visual styling already used by the current RackEditor canvas.

This document is **specification only**. It does not define implementation details yet.

---

## 2. Canonical Export Format

### 2.1 Required export format
The canonical export format shall be:

- **DXF (Drawing Exchange Format)**
- **ASCII DXF** for maximum interoperability
- **AutoCAD DXF 2018 or later-compatible structure**

### 2.2 Why DXF
DXF is selected because it is:

- widely supported by CAD tools,
- text-based and inspectable,
- easier to generate than DWG,
- suitable for 2D plan drawings,
- and appropriate as the initial interoperable CAD exchange format.

### 2.3 Non-goals for the first export specification
The first export specification does **not** require:

- native DWG generation,
- 3D solids,
- paper-space layouts,
- plotting configuration,
- or BIM semantics.

If later needed, DWG/PDF exports can be generated from the DXF-compatible geometry model.

---

## 3. Applicable Standards and Conventions

The export shall align with the following standards and industry conventions where applicable:

### 3.1 File and CAD exchange
- **Autodesk DXF Reference** is the governing file-structure reference for the export.
- The export shall write **ASCII DXF** using the standard tagged-data structure of alternating **group code** and **value** lines.
- The export shall target the **AutoCAD 2018 DXF dialect** and shall therefore write:
  - `$ACADVER = AC1032`
  - `$INSUNITS = 6` (meters)
  - `$MEASUREMENT = 1` (metric)
- The export shall contain these sections in this order:
  1. `HEADER`
  2. `TABLES`
  3. `BLOCKS`
  4. `ENTITIES`
  5. `OBJECTS`
  6. `EOF`
- `CLASSES` may be omitted in the first version because no custom classes are required for interoperability.
- The export shall restrict itself to broadly supported 2D entity types only:
  - `LINE`
  - `LWPOLYLINE`
  - `CIRCLE`
  - `ARC`
  - `ELLIPSE`
  - `HATCH`
  - `TEXT`
  - `MTEXT`
  - `DIMENSION`
  - `INSERT` only if reusable blocks are later introduced
- The export shall not emit proprietary or weakly documented DXF objects such as dynamic blocks, ACIS solids, regions, constraints, or vertical-product custom entities.

### 3.2 Technical drawing presentation
- **ISO 128** is adopted for line presentation and view clarity using the following project rules:
  - visible object edges: continuous lines
  - hidden features: dashed lines and only when required to communicate geometry
  - center/origin references: center lines
  - section or hatch indication: thin graphical patterning
  - stronger visual importance shall always be expressed by thicker lineweight, never only by color
  - the drawing shall remain readable in monochrome plotting
- **ISO 129-1** is adopted for dimensioning using the following project rules:
  - the same dimension shall not be given more than once
  - dimensions shall be placed on the clearest view that describes the feature
  - hidden lines shall not be dimensioned unless no other view is possible
  - dimensions shall describe the object, not be inferred by scaling the drawing
  - diameter dimensions shall use the diameter symbol `⌀`
  - radius dimensions shall use the prefix `R`
  - angular dimensions shall be expressed in degrees
- **ISO 5455** is adopted for approved paper-space scales. If sheet layouts are introduced later, only these scales may be used:
  - full size: `1:1`
  - enlargements: `2:1`, `5:1`, `10:1`
  - reductions: `1:2`, `1:5`, `1:10`, `1:20`, `1:50`, `1:100`, `1:200`, `1:500`, `1:1000`
- **ISO 5457** is adopted for sheet sizes. If sheet layouts are introduced later, only these base trimmed sizes may be used:
  - `A4 = 210 × 297 mm`
  - `A3 = 297 × 420 mm`
  - `A2 = 420 × 594 mm`
  - `A1 = 594 × 841 mm`
  - `A0 = 841 × 1189 mm`
- **ISO 7200** is adopted for title-block data fields. If title blocks are introduced later, the mandatory fields shall be:
  - title
  - created by
  - approved by
  - legal owner
  - document type
  - drawing number
  - sheet number / number of sheets
  - date of issue
- **ISO 13567** is adopted only as an organizational principle for separating element, presentation, and status. The first version uses a fixed project layer profile rather than full free-form ISO 13567 coding.

### 3.3 Units and measurement conventions
- Metric drafting conventions shall be used as the baseline.
- Geometry shall be exported in **meters** because the current editor uses:
  - **1 world unit = 1 meter**.
- All coordinates shall be written as decimal metric values on the XY plane.
- All angles shall be written in decimal degrees.
- All model geometry shall be authored at true size, meaning **1 drawing unit = 1 meter** in the exported DXF.
- If a displayed dimension is written into the drawing, its numeric text shall also be metric by default.

### 3.4 When no formal standard dictates appearance
For colors, fills, visual hierarchy, and object styling that are not strictly imposed by a CAD standard, the export shall follow the **current RackEditor light-theme CAD canvas styling**.

Dark mode is considered a UI theme only and shall **not** define the CAD export appearance.

## 4. Current RackEditor Defaults That Must Be Preserved

These current editor rules are part of the export baseline.

### 4.1 Coordinate system
- 2D Cartesian plane
- `X` increases to the right
- `Y` increases downward in the current canvas world, but exported CAD geometry shall use **positive Y upward**
- therefore, the export shall map editor world coordinates to CAD coordinates using a consistent axis conversion so the drawing opens in a conventional CAD orientation
- All exported entities shall lie on **Z = 0**

### 4.2 World units
- **1 world unit = 1 meter**
- Internal/exported geometry shall therefore be written in meters

### 4.3 Current measurement formatting behavior
The current editor formats values adaptively for display:

- `>= 1000 m` → km
- `>= 1 m` → m
- `>= 0.01 m` → cm
- `< 0.01 m` → mm

For CAD export, this adaptive formatting applies only to labels/dimensions when needed. Geometry itself remains stored in **meters**.

### 4.4 Current drawing context defaults
The current canvas uses:

- adaptive grid,
- rulers,
- origin crosshair,
- light/dark themes,
- entity fills/strokes for racks, walls, columns, and text notes.

Only actual drawing entities shall be exported by default. UI-only aids shall be excluded unless a future option explicitly enables them.

## 5. Required DXF Compatibility Rules

To ensure the file can be opened by CAD systems, the exported DXF must satisfy the following.

### 5.1 Minimum DXF sections
The file shall contain valid DXF structure including at least:

- `HEADER`
- `TABLES`
- `BLOCKS`
- `ENTITIES`
- `OBJECTS`
- `EOF`

### 5.2 Required header information
The header shall define, at minimum:

- drawing version
- insertion units
- measurement mode
- extents
- text style defaults

Required values:

- `$ACADVER = AC1032`
- `$INSUNITS = 6` for meters
- `$MEASUREMENT = 1` for metric
- `$EXTMIN` derived from drawing bounds minimum XY
- `$EXTMAX` derived from drawing bounds maximum XY
- `$LUNITS = 2` for decimal linear units
- `$AUNITS = 0` for decimal degrees

### 5.3 Required tables
The DXF shall define the tables needed by the entities used:

- `LAYER`
- `LTYPE`
- `STYLE`
- `DIMSTYLE` if dimensions are exported
- `APPID` if application metadata/XDATA is exported

The following minimum table content is mandatory:

- `LTYPE` shall define at least:
  - `CONTINUOUS`
  - `CENTER`
  - `HIDDEN`
- `STYLE` shall define at least one text style named `RACKEDITOR_STD`
- `DIMSTYLE` shall define at least one style named `RACKEDITOR_DIM` if dimensions are exported
- `APPID` shall define:
  - `ACAD`
  - `RACKEDITOR` if any XDATA is written

### 5.4 Entity compatibility rules
To maximize interoperability:

- use only the entity types listed in section 3.1
- use `BYLAYER` for color, linetype, and lineweight unless an entity type requires an explicit override
- keep all geometry planar at `Z = 0`
- use closed `LWPOLYLINE` entities for rectangles and other closed planar outlines
- use `CIRCLE` for true round columns and `ELLIPSE` only when the shape is not circular
- use `HATCH` only with solid fill or simple ANSI-compatible hatch patterns
- avoid splines entirely in the first version
- avoid proxy graphics and annotative objects in the first version

### 5.5 Text encoding rules
For cross-platform compatibility:

- visible drawing text shall be restricted to printable ASCII in the first version unless a later implementation validates broader encoding support
- unsupported characters shall be transliterated rather than silently dropped
- single-line labels shall use `TEXT`
- multi-line notes shall use `MTEXT`
- line breaks shall only be emitted inside `MTEXT`

## 6. Export Scope

### 6.1 Entities that must be exportable
The export must support these current entity categories:

- rack modules
- rack lines
- walls
- columns
- text notes

### 6.2 Entities that shall not be exported by default
These are UI aids, not design geometry:

- adaptive grid
- rulers
- corner ruler patch
- zoom HUD
- cursor coordinate HUD
- selection rectangle
- drag previews / ghost geometry
- temporary snapping guides
- selection highlights

### 6.3 Future optional export content
These may be added later as export options:

- dimensions
- title block
- sheet border
- reference grid
- origin marker
- validation callouts
- layer states
- semantic XDATA for rack domain objects

## 7. Geometric Rules

### 7.1 Global geometric rules
All exported geometry shall:

- be written at **1:1 model scale**,
- use meters as the geometric unit,
- preserve the current layout coordinates,
- preserve entity rotation,
- preserve entity extents,
- avoid duplicate coincident geometry where a shared physical element is intended,
- and remain valid when opened in standard CAD applications.

### 7.2 Precision
Required numeric precision:

- store coordinates with at least **millimeter-level precision**,
- serialized precision shall be **0.001 m** or better,
- do not round so aggressively that rack adjacency or wall thickness changes visually.

### 7.3 Rotation
- Rotation shall be preserved in degrees.
- For 2D export, rotations shall be around the entity insertion/base point on the XY plane.

### 7.4 Bounding extents
The exported drawing shall compute extents from all visible exported entities.

## 8. Layering Standard

Layering shall be used so the file is usable immediately in external CAD systems.

### 8.1 Layer naming approach
The first version shall use a **fixed project layer profile** rather than free-form names.

Hard rules:

- Layer names shall use only uppercase letters, digits, and underscore.
- Layer names shall be limited to **31 characters maximum**.
- Each semantic object class shall map to exactly one predefined layer.
- The exporter shall create only the layers listed in section 8.2 unless a future version of this specification adds more.
- This fixed layer profile implements the ISO 13567 principle of separating element categories and presentation categories, but does not expose full ISO 13567 field coding in the first version.

### 8.2 Required layers
At minimum, the file shall define these layers:

| Layer | Purpose | Color | Linetype | Lineweight | Plot by default |
|---|---|---|---|---:|---:|
| `RACK_OUTLINE` | Outer rack perimeter / visible rack extents | `#1d4ed8` | `CONTINUOUS` | 0.35 mm | Yes |
| `RACK_FRAME` | Upright/frame members | `#3b82f6` | `CONTINUOUS` | 0.25 mm | Yes |
| `RACK_BEAM` | Beam members | `#fb923c` | `CONTINUOUS` | 0.25 mm | Yes |
| `RACK_LABEL` | Rack labels and identifiers | `#1e40af` | `CONTINUOUS` | default | Yes |
| `WALL` | Wall geometry | `#6b7280` | `CONTINUOUS` | 0.30 mm | Yes |
| `COLUMN` | Column geometry | `#4b5563` | `CONTINUOUS` | 0.25 mm | Yes |
| `ANNOTATION_TEXT` | Free text notes | `#1f2937` | `CONTINUOUS` | default | Yes |
| `ANNOTATION_DIM` | Dimensions if exported | `#374151` | `CONTINUOUS` | 0.18 mm | Yes |
| `AUX_ORIGIN` | Optional origin indicator | `#94a3b8` | `CENTER` | 0.18 mm | No by default |
| `AUX_GRID` | Optional exported reference grid | `#c5c8ce` | `CONTINUOUS` | 0.13 mm | No by default |
| `METADATA` | Optional non-plot helper entities | `#9ca3af` | `CONTINUOUS` | default | No by default |

### 8.3 Layer behavior
Each entity shall:

- belong to exactly one primary layer,
- use `BYLAYER` color, linetype, and lineweight,
- avoid mixing object categories on the same layer,
- use non-plot layers only for optional helper geometry or metadata carriers,
- and never place production geometry on `AUX_*` or `METADATA` layers.

## 10. Line Types and Line Weights

The export shall follow a clear drafting hierarchy consistent with ISO 128 principles.

### 10.1 Required linetypes
At minimum, the DXF linetype table shall define and use:

- `CONTINUOUS` for visible geometry, text leaders, and dimension lines
- `HIDDEN` for hidden features if they are ever exported
- `CENTER` for origin axes, center marks, or optional reference axes

Project rule:

- racks, walls, columns, labels, and text notes shall use `CONTINUOUS`
- `HIDDEN` shall not be used in the first version unless a future export adds hidden-detail views
- `CENTER` shall only be used on `AUX_ORIGIN` or future centerline/reference layers

### 10.2 Required lineweight hierarchy
If lineweights are written, they shall use this hierarchy:

| Category | Required lineweight |
|---|---:|
| Major visible outlines | 0.35 mm |
| Secondary object edges | 0.25 mm |
| Internal details | 0.18 mm |
| Hatches / auxiliary guides | 0.13 mm |
| Dimensions / annotation leaders | 0.18 mm |
| Text stroke behavior | CAD default |

Only these weights shall be used in the first version.

### 10.3 Mapping to current editor objects
Mandatory mapping:

| Entity content | Lineweight |
|---|---:|
| Rack outer border | 0.35 mm |
| Rack frames | 0.25 mm |
| Rack beams | 0.25 mm |
| Walls | 0.30 mm |
| Columns | 0.25 mm |
| Column internal X | 0.18 mm |
| Text notes | default/by style |
| Optional grid | 0.13 mm |
| Optional origin marker | 0.18 mm |

## 11. Measurement and Dimensioning Rules

### 11.1 Base unit for geometry
The exported file shall use:

- **meters** as the geometric unit,
- with model geometry written 1:1.

### 11.2 Display unit for annotations and dimensions
For the first export specification:

- metric projects shall display dimensions in **meters** by default
- linear dimensions shall use decimal notation
- precision shall default to **0.001 m**
- angular dimensions shall use decimal degrees with precision **0.1°**
- trailing zeros may be suppressed only if doing so does not remove the unit interpretation or reduce clarity

### 11.3 Relationship to current catalog dimensions
Rack catalog semantics may originate from imperial source values such as:

- frame depth in inches,
- frame height in inches,
- beam length in inches,
- hole spacing in inches.

However:

- exported plan geometry shall still be converted and written in **meters**,
- original source values may optionally be stored as metadata.

### 11.4 Dimension style requirements
If dimensions are exported, the dimension style shall define at least:

- linear unit format,
- precision,
- text height,
- arrow size,
- extension line offset,
- extension beyond dim line,
- gap around text,
- and overall scale behavior.

Required default dimension style `RACKEDITOR_DIM`:

- unit format: decimal
- unit: meters
- precision: 0.001
- angular precision: 0.1°
- annotation color: by layer on `ANNOTATION_DIM`
- lineweight: 0.18 mm
- text height on plotted output: 2.5 mm
- arrow size on plotted output: 2.5 mm
- extension line offset from object: 1.5 mm
- extension line overshoot beyond dimension line: 1.5 mm
- gap around dimension text: 0.8 mm
- dimension text aligned with the dimension line for aligned dimensions

Additional ISO 129-1 drafting rules to follow whenever dimensions are exported:

- do not duplicate a dimension in multiple places
- place dimensions outside the object whenever practical
- dimension from true geometric references, not from hatch edges or visual fills
- do not dimension to hidden lines
- use ordinate dimensions only if a future view requires coordinate-based annotation from a declared origin

## 12. Text and Annotation Rules

### 12.1 Text entity types
Use:

- `TEXT` for simple short labels,
- `MTEXT` for notes or multi-line annotations.

### 12.2 Text content rules
Text shall preserve:

- rack labels,
- user text notes,
- optional dimension text,
- optional metadata notes if visibly exported.

### 12.3 Text style
Define at least one default text style.

Required default text style `RACKEDITOR_STD`:

- font family: CAD-safe sans-serif fallback
- preferred font mapping: `Arial` or CAD default simplex-style sans-serif equivalent
- width factor: `1.0`
- oblique angle: `0`
- mirrored text: not allowed
- upside-down text: not allowed
- no dependency on a proprietary font

If the current UI font is unavailable in CAD, use a standard fallback such as:

- Arial,
- Helvetica-like,
- or the CAD application's default sans-serif equivalent.

### 12.4 Text height
Text height shall be preserved from entity semantics where explicitly defined.

If no explicit export rule exists for a label type, these plotted text heights shall be used:

- rack labels: 3.5 mm
- text notes: 2.5 mm
- dimension text: 2.5 mm
- optional sheet title text: 5.0 mm

All visible text shall remain horizontal and readable from the bottom or right side of the sheet if paper space is introduced later.

## 13. Object-by-Object Export Mapping

### 13.1 Rack modules and rack lines
These shall export as visible 2D plan geometry.

Minimum required representation:

- outer rack boundary,
- frame/upright geometry,
- beam geometry,
- label text if present.

Required CAD mapping for the first version:

- outer boundary: `LWPOLYLINE`
- internal frame members: `LWPOLYLINE`
- beam members: `LWPOLYLINE`
- optional fills: `HATCH`
- labels: `TEXT` or `MTEXT`

### 13.2 Shared rack uprights
The current editor visually suppresses duplicate shared uprights between adjacent rack bays.

The export must preserve the same physical drafting intent:

- do not duplicate shared members if they represent the same physical frame,
- geometry shall remain clean for downstream CAD use.

### 13.3 Walls
Walls shall export as linear plan elements with thickness.

Required mapping for the first version:

- closed `LWPOLYLINE` rectangle for wall body,
- optional `HATCH` for fill,
- optional patterned hatch for dotted texture look.

Required wall information:

- start/base location,
- length,
- thickness,
- rotation,
- label if any.

### 13.4 Columns
Columns shall export as structural obstacles.

For rectangular columns:

- closed `LWPOLYLINE`
- optional solid `HATCH`
- internal diagonal `LINE`s forming an X

For round columns:

- `CIRCLE` or `ELLIPSE`
- optional solid `HATCH`
- internal diagonal cross if visually supported

Required column information:

- center point,
- width,
- depth,
- shape,
- label if any.

### 13.5 Text notes
Text notes shall export as annotation entities.

Required note information:

- insertion point,
- rotation,
- content,
- text height.

Required mapping for the first version:

- `TEXT` for single-line notes,
- `MTEXT` for multi-line notes.

## 14. Metadata Requirements

To support traceability and future semantic workflows, the export should carry metadata where possible.

### 14.1 File-level metadata
Recommended metadata:

- application name: `RackEditor`
- export timestamp
- export format version
- project name / drawing name if available
- units
- drawing bounds
- app version if available

### 14.2 Entity-level metadata
Where possible, preserve identifiers such as:

- layout entity id
- entity type
- linked rack domain id
- label/name
- source dimensions

### 14.3 Metadata carrier options
Metadata can be stored through one of these mechanisms:

- DXF XDATA,
- extension dictionaries,
- hidden/non-plot text only if absolutely necessary,
- or a companion file in a later phase.

For the first export design, metadata is recommended but secondary to CAD interoperability.

## 15. Layout and Sheet Rules

### 15.1 Model space
The first export shall be **model-space first**.

That means:

- all geometry is written at 1:1,
- no paper-space layout is required,
- no title block is required,
- no viewport setup is required.

### 15.2 Future sheet support
If sheet export is later added, it shall follow:

- ISO 5457 sheet sizes only: `A4`, `A3`, `A2`, `A1`, `A0`
- default orientation:
  - `A4` portrait
  - `A3`, `A2`, `A1`, `A0` landscape
- title block located in the bottom-right corner
- ISO 7200 mandatory title-block fields:
  - title
  - created by
  - approved by
  - legal owner
  - document type
  - drawing number
  - sheet number / total sheets
  - date of issue
- only ISO 5455 approved scales listed in section 3.2 may be used
- model geometry shall remain true size in model space; scaling shall occur only through viewport or sheet presentation, never by scaling the exported model entities themselves

### 15.3 Origin and placement
Default behavior:

- preserve the editor world coordinates,
- use the current drawing origin as the CAD origin reference,
- do not re-center geometry unless the export mode explicitly requests normalization.

## 16. Validation Requirements Before a File Is Considered Valid

A generated export file must satisfy all of the following.

### 16.1 Structural validation
- valid DXF section order
- valid table definitions for all used layers/styles/linetypes
- valid end-of-file marker
- no missing references to layers, linetypes, or styles

### 16.2 Geometric validation
- no malformed closed polylines
- no NaN or infinite coordinates
- no unintended Z offsets
- no invalid negative thickness/size values
- rotations serialized consistently

### 16.3 Visual validation
When opened in a CAD tool, the drawing must:

- show correct object positions,
- show correct sizes,
- preserve rotation,
- preserve readable layer-based styling,
- and remain understandable without relying on the RackEditor UI.

### 16.4 Interoperability validation
The file shall open without repair prompts in common CAD software classes such as:

- AutoCAD-compatible viewers,
- DraftSight-class tools,
- LibreCAD/QCAD-class tools where supported by entity set,
- and cloud CAD viewers that support standard DXF.

## 17. Current Rack-Domain Context Relevant to Export

The current product models selective pallet racking and already uses domain constraints that matter for exported documentation.

Relevant current facts include:

- rack drawings are a projection of the domain model, not the source of truth,
- frame hole spacing is based on **2 inch increments**,
- default rack setup is based on common selective rack values such as:
  - 96 in beam width,
  - 42 in frame depth,
  - 144 in frame height,
- those source dimensions are already converted to metric for layout geometry.

This means the CAD export should prioritize:

- exact metric plan geometry,
- optional preservation of source semantic values as metadata,
- and a clean drafting representation of rack structures.

---

## 18. Required Defaults for the First Version

Unless an explicit export option overrides them, the first version shall default to:

- format: **ASCII DXF**
- geometry space: **model space only**
- unit: **meters**
- scale: **1:1**
- Z coordinate: **0**
- styling basis: **current RackEditor light theme**
- colors/lineweights: **BYLAYER**
- exported content: **racks, walls, columns, text notes**
- excluded content: **grid, rulers, selection aids, temporary previews**
- precision: **0.001 m or better**

## 19. Out-of-Scope for This Specification Version

This document does not yet require:

- 3D rack elevations,
- structural engineering calculation output,
- manufacturing fabrication drawings,
- automatic title blocks,
- paper-space layouts,
- PDF plotting rules,
- DWG-native writing,
- or BIM/IFC export.

## 20. Final Acceptance Criteria for the Future Export Feature

A CAD export implementation shall be considered compliant with this specification when it produces a file that:

1. opens successfully in a standard CAD application,
2. preserves all visible layout geometry for supported entities,
3. writes geometry in meters at 1:1 scale,
4. uses valid layers, styles, and compatible DXF entities,
5. follows drafting conventions from established CAD standards where applicable,
6. follows current RackEditor visual styling where standards do not dictate appearance,
7. excludes UI-only editing aids by default,
8. and is readable, organized, and usable for downstream CAD workflows.
